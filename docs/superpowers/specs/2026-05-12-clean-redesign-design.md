# Tigerbookmaker 깔끔하게 정리 — 설계안

**작성일**: 2026-05-12
**상태**: 사용자 검토 대기
**예상 작업량**: ~6일 (5개 PR로 분할)

---

## 1. 배경 · 목적

베타 운영 결과 책 1권을 끝까지 완성한 사용자가 아직 없음 (CEO 본인 외 0건). 그럼에도 사이트는 라우트 87개, UI 페이지 30+개, 60+ 기능을 가지고 있어 **첫 책 완성 흐름이 사이드 기능에 가려져 있음**.

이 spec은 사이트를 다음 4가지 방향으로 동시 재정리한다.

1. **기능 cut**: 87 라우트 → 13개, 60+ 기능 → 18개. 첫 책 1권 완성에 직접 기여하는 것만 살림.
2. **디자인 일관성**: 6 컬러 토큰 / 5 타이포 / 3 컴포넌트 통일.
3. **모바일 우선**: 핵심 5페이지 phone-first 손질.
4. **이미지 군더더기 제거**: 표지 템플릿 7→3, mock 박스 제거, 이모지 페이지당 ≤2.

타겟 페르소나는 **직장인 부수익러** (크몽에서 ₩30k~₩100k에 책 판매하려는 30대 직장인). 이 페르소나가 "30분에 1권"을 검증할 수 있는 최소 사이트가 목표.

## 2. 범위

### 2.1 포함 (v1)

**기능 정리**:
- 60+ 기능 → 18개로 cut/defer
- 30+ UI 페이지 → 13개로 cut/defer
- DB 스키마는 그대로 유지 (defer = 코드/라우트 살려두고 메뉴 노출만 제거)

**디자인 시스템**:
- 6 컬러 토큰 확정 + `tailwind.config.ts` 정리
- 3 공통 컴포넌트 추출 (`<Button>`, `<Card>`, `<Section>`)
- 13 페이지를 동일 패턴(`Eyebrow + H2 + 설명문`)으로 재정렬

**모바일**:
- 햄버거 메뉴 + 스크롤 시 헤더 숨김 (`components/Header.tsx`)
- 핵심 5 페이지 phone-first 검토 + 타이포 스케일 조정

**이미지**:
- 샘플 표지 SVG 7→3 템플릿
- 기능 박스 미니 mock 제거 → 실제 결과 1장 또는 텍스트만
- 페이지별 의미 있는 스켈레톤 (현재 "로딩 중..." 7페이지 동일)
- 이모지 페이지당 최대 2개로 제한

**Retention 시드**:
- 이메일 리커버리 시퀀스 (가입 후 책 안 만든 사용자, 자료 업로드 후 멈춘 사용자 등 3 트리거)

### 2.2 미포함 (Out of scope)

- **DB 스키마 변경**: 시리즈/챌린지/트렌드 테이블 다 유지. 메뉴 노출만 제거.
- **결제 플로우 재설계**: P0 #4 (무료 미리보기 게이트)는 별도 spec.
- **AI 모델·프롬프트 튜닝**: 책 품질 자체는 손대지 않음.
- **새 기능 추가**: 이메일 리커버리 외 새 기능 없음. 정리만.
- **외부 채널 출판/링크 인 바이오 페이지 제거**: 라우트는 살려두되 네비/홈에서만 숨김.
- **A/B 테스트 / Analytics 변경**: 인프라 그대로.

## 3. 주요 설계 결정 + 이유

### 3.1 기능 평가 결과 (Keep / Defer / Cut)

#### ✅ KEEP (18 기능, 13 페이지) — 첫 책 1권 완성에 직접 기여

| 카테고리 | 기능 | 페이지/API |
|---------|------|-----------|
| 입력 | 주제 + 자료 입력 | `/new` |
| 입력 | 주제 제안 (막힘 해소) | `/api/generate/topic-suggestions` |
| 입력 | 톤 추천 (인터뷰 마찰 ↓) | `/api/generate/tone-recommend` |
| 자료 | RAG 업로드 (PDF/URL/text) + chunks | `/api/reference/*` |
| 자료 | 자료 요약 | `/api/generate/reference-summary` |
| 자료 | AI 인터뷰 질문 | `/api/generate/interview-question` |
| 본문 | 목차 생성 | `/api/generate/toc` |
| 본문 | 챕터 본문 + 이어쓰기 + 편집 | `/api/generate/chapter*` |
| 본문 | 워크스페이스 | `/write`, `/write/setup` |
| 표지 | 컨셉 + 헤드라인 + 베리에이션 + 오버레이 | `/api/generate/cover-*` |
| 끝마무리 | 미리보기 + 내보내기 (PDF/EPUB/DOCX) | `/preview`, `/export` |
| 관리 | 내 책 목록 | `/projects` |
| 결제 | 충전 + 잔액 | `/billing`, `/usage` |
| 인증 | 회원가입 / 로그인 / 비번 재설정 | `/login`, `/reset-password`, `/api/auth/*` |
| 공유 | 책 마케팅 페이지 (작은 retention 시드) | `/book/[id]` |
| Retention | **이메일 리커버리 시퀀스 (신규)** | 기존 `/api/cron/email-sequence` 확장 |
| 마케팅 | 랜딩 | `/` |
| 법적 | 약관·개인정보·환불 | `/legal/*` |

총 페이지: 9 핵심 + 3 법적 + `/pricing` = **13개**.

**Retention은 "이메일 리커버리 시퀀스" 1개에 집중하는 이유**: A=0 단계에선 다른 retention 기능(시리즈/챌린지/트렌드 등)이 다 "사용자 풀이 있어야 작동"하는 구조. 이메일 리커버리는 사용자 1명만 있어도 작동. 현재 단계에서 90% 가치 만듦.

#### 🟡 DEFER (메뉴 노출 제거, 라우트는 유지)

| 기능 | 켜는 트리거 |
|------|----------|
| 시리즈 (`/series`) | 첫 권 완성 사용자 5명+ |
| 챌린지 (`/challenges`) | 활성 사용자 50명+ |
| 트렌드 (`/trends`) | 누적 책 100권+ |
| 추천 보상 (`/r/[code]`, referral) | NPS > 50 |
| 책 채팅 (`/book/[id]/chat`) | 첫 책 완성 후 별도 평가 |
| 작가 구독 / 알림 | 작가 5명+ 활성 |
| 책 리뷰 / 책 구독 | 독자 트래픽 발생 후 |
| 인포그래픽 / 슬라이드 / 오디오북 / 미리보기 영상 / 번역 | 책 본체 완성 사용자 누적 후 |
| Repurpose 5종 (블로그/이메일/인스타/카카오/유튜브) | 첫 책 완성 후 |
| A/B 테스트 | 트래픽 발생 후 |
| 작가 뱃지 / 작가 스탯 | 활성 작가 5명+ |
| Meta 광고 패키지 | 광고 돌릴 사용자 등장 후 |

#### ❌ CUT (메뉴/홈/네비에서 완전 제거 — 라우트만 dead-end)

| 기능 | 왜 cut |
|------|-------|
| `/external-publishing` | 부수익러는 크몽 → 끝. Amazon KDP/리디/교보/알라딘은 다른 페르소나. |
| `/external-linkbio` | 책 1권도 없는데 인스타 bio 정리 의미 X |
| `/import-blog` | `/new`와 진입 혼선. 첫 책 흐름은 1개여야 함 |
| `/share/[id]` 별도 페이지 | `/book/[id]`와 중복 |
| `/kmong-listing-helper` | DEFER로 분류했지만 첫 책 완성 후 가치. 지금은 cut. |
| `/u/[handle]` 작가 프로필 | 작가 본인 1명. 의미 X |

### 3.2 디자인 토큰

```
tiger-orange   #f97316   CTA, 강조, 라이브 신호
ink-900        #171717   본문 텍스트, 헤딩
gray-600       #525252   서브 텍스트
gray-200       #e5e5e5   디바이더, 카드 보더
bg-base        #fafafa   페이지 배경
bg-elevated    #ffffff   카드 배경
```

**왜 6개로 고정**: 토큰이 늘면 페이지마다 "이번엔 어떤 컬러?" 결정이 생김 → 일관성 깨짐. 6개면 새 페이지 만들 때 무조건 이 6개에서만 고름 → 디자인 결정 99% 사라짐. 샘플 표지 안의 다양한 컬러(에세이 stone, 매뉴얼 cyan 등)는 **표지 SVG 안에서만** 살리고 UI 외부엔 절대 노출 X.

**타이포 스케일**:
```
hero         text-6xl md:text-7xl    font-black tracking-tight
h2 섹션      text-4xl md:text-5xl    font-black tracking-tight
h3 본문 헤딩 text-2xl                  font-bold
body         text-base md:text-lg    text-gray-600
caption      text-xs font-mono uppercase tracking-[0.2em] text-tiger-orange
```

**섹션 패턴**: 모든 페이지 헤더는 `Eyebrow(주황 caption) + H2(검정 헤딩) + 짧은 설명문` 3단 구조. 13 페이지가 같은 리듬으로 흐름.

### 3.3 공통 컴포넌트 3개

| 컴포넌트 | 변형 | 위치 |
|---------|-----|------|
| `<Button>` | `primary` (주황) / `secondary` (테두리만) / `ghost` (텍스트만) | `components/ui/Button.tsx` (신규) |
| `<Card>` | 단일 변형: `border border-gray-200 rounded-2xl p-6` | `components/ui/Card.tsx` (신규) |
| `<Section>` | 단일 변형: `max-w-6xl mx-auto px-6 py-24 md:py-32` + `border-t` divider | `components/ui/Section.tsx` (신규) |

**왜 3개로 제한**: 4번째 변형이 추가되는 순간 일관성 깨짐. 13개 페이지가 이 3개 컴포넌트 조합으로만 만들어지면 디자인 시스템이 자동 유지됨.

**CTA 텍스트 통일**: 모든 페이지의 메인 CTA는
```
무료로 시작 — ₩5,000 크레딧 받기 →
```
파생(충전 등)에서만 텍스트 교체.

### 3.4 모바일 — 햄버거 + 스크롤 시 헤더 숨김

| 패턴 | 선택 | 이유 |
|------|------|------|
| 네비게이션 | 햄버거 (햄버거 아이콘 → 풀스크린 메뉴) | 핵심 페이지 9개라 하단 탭바(표준 4~5개) 못 씀. 햄버거는 모바일 표준 |
| 헤더 동작 | 스크롤 다운 시 숨김, 업 시 등장 | `/write`, `/preview` 같은 콘텐츠 페이지에서 헤더가 화면 14% 차지. 콘텐츠 우선 |
| 터치 타겟 | 최소 높이 44px | iOS HIG 표준 |
| 폼 입력 | 한 화면 한 단계 (Step 1만 보이고 Step 2/3는 다음 화면) | `/new`가 현재 한 화면에 다 들어가서 스크롤 부담 |

**핵심 5 페이지 모바일 우선순위**:
- 🔴🔴🔴 `/` (광고 도착지)
- 🔴🔴🔴 `/new` (가입 후 첫 화면)
- 🔴🔴 `/write` (자료 업로드·인터뷰)
- 🔴🔴 `/preview` (책 확인)
- 🔴 `/export` (보통 PC에서)

### 3.5 이미지 정리

| 변경 | 결과 | 이유 |
|------|------|------|
| 샘플 표지 7→3 템플릿 | 단색 미니멀(Penguin) / 잡지(Monocle) / 그라데이션 | 7장르 7디자인 = 시각 잡음. 카테고리별 매핑하면 일관성 ↑. 크몽 썸네일은 작은 사이즈라 단순 표지가 클릭률 ↑ |
| 기능 박스 미니 mock 제거 | 텍스트 + 실제 결과 스크린샷 1장 | mock은 정보 0, 장식만. 실제 스크린샷 1장이 신뢰감 ↑ |
| 이모지 페이지당 ≤2 | 헤더·CTA 액센트만 | 현재 9개 이모지 남발 → 가독성 ↓. 2개 제한이면 진짜 강조할 거만 살아남음 |
| "로딩 중..." → 의미 있는 스켈레톤 | 페이지별 다른 회색 박스 (책 카드, 챕터 배치 등) | SSR로 보이기 때문에 SEO·공유 시 빈 페이지 안 보임 |
| 장식 blob/glow 정리 | 히어로 1개 + 다크 CTA 1개만 | 현재 모든 섹션에 glow → 시각 노이즈. 빈 공간 + 디바이더로 절제 |

**3 표지 템플릿 카테고리 매핑**:
- **단색 미니멀** (Penguin Classics): 실용서, 매뉴얼, 전문서
- **잡지** (Monocle): 자기계발서, 에세이
- **그라데이션 + 큰 타이포**: 재테크, 웹소설

### 3.6 이메일 리커버리 시퀀스 (Retention 시드)

기존 `/api/cron/email-sequence` 인프라 활용. 3 트리거:

| 트리거 | 발송 시점 | 메일 내용 |
|--------|---------|----------|
| 가입 후 책 안 만듦 | 24시간 후 | "₩5,000 크레딧 그대로 있어요. 어떤 책 만들까 막막하면 [주제 제안] 눌러보세요" |

| 챕터 5개 이상 만들고 멈춤 | 7일 후 | "${title} 책 거의 다 만들어졌어요. 나머지 챕터 자동 생성하시겠어요?" |
| 책 완성 (export) | 즉시 | "축하해요. 다음 책 만드실 분들 위해 후기 한 줄 부탁드려요" + 추천 보상 안내 |

DB 추가 X — `users` + `book_projects` + `balance_transactions` 시간 비교로 충분.

### 6.1 리스크

| 리스크 | 영향 | 완화 |

## 7. 의사결정 요약

| 결정 | 선택 | 핵심 근거 |
